## 2. What Is Cybersecurity, Really?

Yaar, pop culture ne cybersecurity ko ek myth bana diya hai — hoodies, dark rooms, Matrix-style green code. **Reality alag hai.**

### A proper definition

> **Cybersecurity** is the discipline of protecting **information systems, the data they process, and the people who depend on them** from harm caused by **adversaries** — whether those adversaries are external attackers, malicious insiders, careless users, or even buggy software acting in unintended ways.

Is definition ko unpack karte hain:

#### "Discipline"
Yeh art nahi hai, engineering hai. Jaise tu reliability engineering padhta hai (MTTR, MTBF, SLOs) — security bhi vaisi hi quantitative + process-driven field hai. **Risk math hota hai. Controls testable hote hain. Incidents post-mortemed hote hain.**

#### "Information systems"
Servers, APIs, databases, mobile apps, IoT devices, cloud workloads, CI/CD pipelines, third-party SaaS — sab. Tu jab Spring Boot microservice deploy karta hai AWS pe, woh ek information system hai jisko protect karna hai.

#### "Data they process"
Sirf "stored data" nahi — **data in motion (network)**, **data at rest (disk)**, aur **data in use (RAM, CPU registers)**. Teeno alag threats face karte hain. Yeh distinction Day 45 (Week 7) pe deep karenge.

#### "People who depend on them"
Yahaan **human factor** important hai. Tera customer apna paisa trust kar raha hai. Tera employee apni privacy trust kar raha hai. **Security ek socio-technical problem hai — sirf code nahi.**

#### "Adversaries"
Adversary ka matlab sirf "hacker in Russia" nahi:
- **External attackers** — script kiddies se lekar nation-states
- **Insiders** — disgruntled employee, contractor with too much access
- **Accidents** — junior dev ne `git push --force` kar diya
- **Software bugs** — race conditions, integer overflows, memory corruption

Ek **architect** ka kaam hai system aisa design karna ke yeh saari categories handle ho.

### The two halves of security

Cybersecurity ke broadly do hisse hain — yeh distinction yaad rakh:

| Half | Focus | Examples |
|------|-------|----------|
| **Defensive (Blue Team)** | Build, monitor, defend | SIEM, IDS, threat modeling, secure coding, incident response |
| **Offensive (Red Team)** | Attack to find weaknesses | Pentesting, red teaming, exploit dev, bug bounties |

Tu **defensive architect** ban raha hai primarily — system design karna jisko break karna mushkil ho. Lekin Q3 (Days 241-270) mein offensive bhi seekhega, kyunki **"think like an attacker"** ke bina defense incomplete hai.

## 3. Where the CIA Triad Came From

CIA Triad random framework nahi hai — yeh **1970s ke US military computing research** se nikla. Background samajhna important hai kyunki yeh limitations bhi bata deta hai.

### Historical timeline (short)

- **1973** — Bell-LaPadula model published (US Air Force, Mitre Corp). Focus: **confidentiality** in multi-level secure systems. "No read up, no write down" rule.
- **1977** — Biba model published. Counterpart: **integrity**. "No read down, no write up."
- **1980s** — Industry combined these ideas. Term **"CIA triad"** popularized in security textbooks.
- **1995** — NIST publishes **SP 800-12** formalizing CIA as the foundation of information security.
- **2002** — US Congress passes **FISMA** (Federal Information Security Management Act), legally codifying CIA-based risk categorization.

### Why this history matters

CIA triad **government secrecy mindset** se aaya hai. Iska matlab:
- Yeh **secrecy-first** framework hai — military ke liye confidentiality top priority thi
- Commercial systems mein priorities **flip** ho sakti hain. E-commerce ke liye availability often #1 hota hai (1 hour downtime = $millions lost).
- Modern attacks aise hain jo CIA mein clean fit nahi karte — **non-repudiation**, **authenticity**, **possession** jaisi cheezein. Isi liye **Parkerian Hexad** (Section 9) exist karta hai.

> **Mentor note:** Frameworks tools hain, religion nahi. CIA tujhe **starting checklist** deta hai — har feature pe puchna "C? I? A?" Lekin agar koi requirement teeno mein clean fit nahi karti, tu galat nahi hai — framework incomplete hai.

## 4. Confidentiality — Deep Dive

### Definition (multiple framings)

> **Confidentiality** ensures that information is **disclosed only to those authorized** to access it.

Alternative framings (sab same baat alag angles se):

- **NIST framing:** "Preserving authorized restrictions on information access and disclosure."
- **Practical framing:** "Right data, right person, right time — aur kisi ko nahi."
- **Architect framing:** "Every byte ka ek authorized audience hai. Audience define karna aur enforce karna confidentiality hai."

### Two kinds of confidentiality

| Type | Means | Example |
|------|-------|---------|
| **Data confidentiality** | Data itself protected | Encrypted SSN in database |
| **Privacy** | User control over their own data | GDPR "right to be forgotten" |

Yeh distinction important hai. Tu data encrypt kar sakta hai (data confidentiality ✅) but user ko bata bhi nahi raha tu uska data collect kar raha hai (privacy ❌). **Confidentiality ≠ Privacy** — both matter, alag angles se.

### How confidentiality is enforced — the layered approach

Real systems mein confidentiality **defense in depth** approach se enforce hoti hai. Soch ek concentric circles ki tarah:

```
                  ┌─────────────────────────────────┐
                  │   1. Physical security          │  (datacenter, devices)
                  │  ┌───────────────────────────┐  │
                  │  │  2. Network controls      │  │  (TLS, VPN, firewall)
                  │  │  ┌─────────────────────┐  │  │
                  │  │  │ 3. App authentication│  │  │  (login, MFA)
                  │  │  │  ┌────────────────┐ │  │  │
                  │  │  │  │4. Authorization│ │  │  │  (RBAC, ABAC)
                  │  │  │  │  ┌──────────┐  │ │  │  │
                  │  │  │  │  │5. Data   │  │ │  │  │  (encryption, masking)
                  │  │  │  │  │  encrypt │  │ │  │  │
                  │  │  │  │  └──────────┘  │ │  │  │
                  │  │  │  └────────────────┘ │  │  │
                  │  │  └─────────────────────┘  │  │
                  │  └───────────────────────────┘  │
                  └─────────────────────────────────┘
                            ASSET (data)
```

**Lesson:** Single control kaafi nahi. Agar attacker ek layer crack kar le, agla layer rokta hai. Yeh **defense in depth** principle hai — hum Day 5 pe full deep karenge.

### Concrete enforcement mechanisms

#### 4.1 Access control at the application layer

```csharp
// .NET 8 — confidentiality via authorization
[ApiController]
[Route("api/v1/employees")]
public class EmployeeController : ControllerBase
{
    [HttpGet("{id}/salary")]
    [Authorize(Roles = "HR,Finance,Manager")]   // Confidentiality control #1
    public async Task<IActionResult> GetSalary(int id)
    {
        var requester = User.GetUserId();
        var employee = await _db.Employees.FindAsync(id);

        // Confidentiality control #2: even authorized roles can only see direct reports
        if (User.IsInRole("Manager") && employee.ManagerId != requester)
            return Forbid();

        return Ok(new { employee.Id, employee.Salary });
    }
}
```

Spring Boot equivalent:

```java
@RestController
@RequestMapping("/api/v1/employees")
public class EmployeeController {

    @GetMapping("/{id}/salary")
    @PreAuthorize("hasAnyRole('HR','FINANCE','MANAGER')")
    public ResponseEntity<SalaryDto> getSalary(@PathVariable Long id, Principal p) {
        var employee = repo.findById(id).orElseThrow();
        if (isManager(p) && !employee.getManagerId().equals(currentUserId(p))) {
            throw new AccessDeniedException("Not your report");
        }
        return ResponseEntity.ok(new SalaryDto(employee));
    }
}
```

Notice: **two layers**. Role check + ownership check. Single role check would let any manager see any employee's salary — too coarse.

#### 4.2 Encryption at rest

```sql
-- PostgreSQL with pgcrypto: column-level encryption for PII
CREATE TABLE customers (
    id           BIGSERIAL PRIMARY KEY,
    email        TEXT NOT NULL,
    -- SSN stored encrypted; key lives in KMS, not in DB
    ssn_encrypted BYTEA NOT NULL,
    created_at   TIMESTAMPTZ DEFAULT NOW()
);

-- Application encrypts before insert, decrypts on read with KMS-fetched key
```

Why column-level vs whole-disk encryption? **Disk encryption protects against laptop theft. Column encryption protects against a DBA peeking.** Different threat models — Day 46 deep dive.

#### 4.3 Encryption in transit

```nginx
# NGINX TLS 1.3 — confidentiality on the wire
server {
    listen 443 ssl http2;
    ssl_protocols TLSv1.3;                    # No TLS 1.0/1.1/1.2
    ssl_ciphers TLS_AES_256_GCM_SHA384:TLS_CHACHA20_POLY1305_SHA256;
    ssl_prefer_server_ciphers off;
    add_header Strict-Transport-Security "max-age=63072000; includeSubDomains; preload" always;
    # ...
}
```

#### 4.4 Data masking & tokenization

Different from encryption — used when **partial visibility** required.

```
Real value:    4532-1521-9876-5432
Masked log:    4532-****-****-5432    (PCI-DSS compliant)
Tokenized DB:  tok_aXz9p2Lq          (random token; maps to real value via vault)
```

### Anti-patterns (mat kar yeh)

| Anti-pattern | Why it's bad |
|---|---|
| Putting secrets in source code | Anyone with repo access = anyone with the secret |
| Encrypting with key stored next to ciphertext | Like locking your door and taping the key to it |
| Logging full request bodies | PII, passwords, tokens land in logs reviewed by anyone |
| `SELECT * FROM users WHERE email LIKE ?` returned to client | Over-disclosure — return only what UI needs |
| Trusting client-side filtering | API returned full salary list, JS hid columns. Attacker reads network tab. |

### Common confidentiality failure modes (memorize)

1. **Misconfigured cloud storage** — S3 bucket `Principal: *` (Capital One, 2019)
2. **Broken access control** — IDOR (Insecure Direct Object Reference): `/api/orders/123` → change to `/api/orders/124` → see someone else's order
3. **Verbose error messages** — stack trace exposes DB schema
4. **Side channels** — timing attacks reveal whether username exists
5. **Insider threats** — DBA with read access to PII table
6. **Backup leaks** — production data restored to staging without sanitization

### Backend-dev intuition

> Jab tu code likh raha ho aur ek field "return karu ya nahi" pe doubt aaye — **default deny**. Field add karna easy hai later, leak hua data wapas nahi aata.

## 5. Integrity — Deep Dive

### Definition (multiple framings)

> **Integrity** ensures that information has not been **modified, deleted, or fabricated** in an unauthorized manner, and that its **source can be trusted**.

Alternative framings:
- **NIST framing:** "Guarding against improper information modification or destruction, including ensuring information non-repudiation and authenticity."
- **Practical framing:** "Jo data tu read kar raha hai, kya woh wahi hai jo original tha?"
- **Architect framing:** "Every state change ka ek authorized author hona chahiye, aur change traceable hona chahiye."

### Two kinds of integrity

| Type | Focus | Example |
|------|-------|---------|
| **Data integrity** | Data itself unchanged | Bank balance shows correct value |
| **System/origin integrity** | Source is who it claims to be | Email actually from bank, not phisher |

Origin integrity = **authenticity**. Yeh CIA mein technically Integrity ke andar fit hota hai, lekin Parkerian Hexad isko alag pillar maanta hai (Section 9).

### Why integrity attacks are scarier than confidentiality

Soch:

**Scenario A (Confidentiality breach):** Hacker tera bank balance dekh leta hai. $5000 hai. Embarrassing, but tera paisa intact.

**Scenario B (Integrity breach):** Hacker tera bank balance **change** kar deta hai. Tujhe lagta hai $5000 hai, transactions allow ho rahi hain, but actually $50,000 ka fraud ho chuka hai. Tu detect bhi nahi kar payega — system "consistent" lag raha hai.

> **Industry quote (Bruce Schneier):** "Confidentiality breaches are embarrassing. Integrity breaches are existential."

Real-world example: **2010 Stuxnet** — Iranian centrifuges. Integrity attack. Control system reported normal speeds while centrifuges spun themselves apart. Operators trust gaya tha system pe — wahi trust weapon ban gaya.

### How integrity is enforced

#### 5.1 Cryptographic hashes

```python
# Hash = fixed-length fingerprint. Same input → same hash. Any change → wildly different hash.
import hashlib

content = b"Order #12345: amount=5000"
hash1 = hashlib.sha256(content).hexdigest()
# d4c3b2a1...

tampered = b"Order #12345: amount=50000"
hash2 = hashlib.sha256(tampered).hexdigest()
# completely different hex string
```

But hashes alone don't prove integrity to a remote party — attacker can change both data and hash. Solution: **MAC**.

#### 5.2 Message Authentication Codes (HMAC)

```csharp
// .NET: HMAC = hash + shared secret. Only someone with the key can produce a valid MAC.
using var hmac = new HMACSHA256(sharedSecretKey);
byte[] tag = hmac.ComputeHash(Encoding.UTF8.GetBytes(payload));
// Send (payload, tag). Receiver recomputes tag with same key. Match = integrity OK.
```

Used in: webhook signatures (Stripe, GitHub), API request signing (AWS SigV4).

#### 5.3 Digital signatures (asymmetric)

HMAC requires both parties share secret. **Digital signatures** use private/public keys — sender signs with private, anyone verifies with public. Bonus: **non-repudiation** (sender can't deny they signed it).

Used in: TLS certs, code signing (NuGet, Maven, container images), Git commit signing.

#### 5.4 Database transactions & constraints

```sql
-- ACID transactions = integrity at the data layer
BEGIN;
UPDATE accounts SET balance = balance - 100 WHERE id = 1;
UPDATE accounts SET balance = balance + 100 WHERE id = 2;
COMMIT;  -- Either both happen, or neither. No partial state.

-- Constraints prevent invalid state
ALTER TABLE accounts ADD CONSTRAINT positive_balance CHECK (balance >= 0);
```

#### 5.5 Append-only audit logs

```sql
-- audit_log: NO updates, NO deletes. Only inserts.
CREATE TABLE audit_log (
    id          BIGSERIAL PRIMARY KEY,
    actor_id    BIGINT NOT NULL,
    action      TEXT NOT NULL,
    target      TEXT NOT NULL,
    payload     JSONB,
    occurred_at TIMESTAMPTZ DEFAULT NOW(),
    prev_hash   TEXT,           -- hash of previous row → tamper-evident chain
    row_hash    TEXT NOT NULL
);

REVOKE UPDATE, DELETE ON audit_log FROM application_user;
```

The `prev_hash` chain is **blockchain-lite** — change any row and downstream hashes break. AWS QLDB, Azure SQL Ledger productize this.

#### 5.6 Optimistic concurrency (preventing lost-update bugs)

```http
GET /api/v1/orders/42
< HTTP/1.1 200 OK
< ETag: "v7"
< { "status": "pending", "amount": 500 }

PUT /api/v1/orders/42
> If-Match: "v7"
> { "status": "shipped" }

< HTTP/1.1 200 OK   (if no one else changed it)
< HTTP/1.1 412 Precondition Failed   (if version changed → reject)
```

Yeh **integrity at the API boundary** hai. Two clients race condition mein ek dusre ka update overwrite nahi kar sakte.

### Anti-patterns

| Anti-pattern | Why it's bad |
|---|---|
| Trusting `Content-MD5` from client | Client computed it — attacker computes valid one too |
| Storing audit logs in same DB attackers can write to | First thing they do: delete log entries |
| Using MD5 or SHA-1 for integrity | Both broken — collisions feasible |
| Skipping `WHERE id = ?` clause in dev "for speed" | One missed `WHERE` = mass data corruption |
| Hand-rolling crypto signatures | Edge cases (padding, timing) break in subtle ways. Use libsodium / well-tested libs. |

### Backend-dev intuition

> Har destructive operation (UPDATE/DELETE) ko **audit log + idempotency key + authorization check** se gate kar. Three layers — single mistake catastrophic na ho.

## 6. Availability — Deep Dive

### Definition

> **Availability** ensures that information and systems are **accessible and usable** by authorized parties **when needed**.

Note the precision: **"when needed"** — not 24/7 in absolute sense. SLAs define acceptable downtime.

### The "uptime" misconception

Junior devs think availability = uptime %. **Architects know better.** Availability includes:

| Dimension | Question | Example failure |
|---|---|---|
| **Uptime** | Is the service running? | Process crashed |
| **Reachability** | Can users get to it? | DNS poisoned, network partition |
| **Performance** | Is it responding fast enough? | DB query takes 30s; users abandon |
| **Capacity** | Can it serve all who need it? | DDoS exhausts connection pool |
| **Correctness under load** | Does it still work right when busy? | Race condition only appears at 10K RPS |
| **Recoverability** | Can it come back after failure? | Backups exist but never tested |

**All five** matter. Tu sirf "uptime monitoring" pe rely kar raha hai — tu blind hai.

### Common availability threats

#### 6.1 DDoS (Distributed Denial of Service)

Layer-by-layer attack categorization:

| OSI Layer | Attack | Example | Defense |
|---|---|---|---|
| L3/L4 (network) | Volumetric — flood bandwidth | UDP flood, amplification | Anycast, scrubbing (Cloudflare, AWS Shield) |
| L4 (transport) | Protocol — exhaust state tables | SYN flood, Slowloris | SYN cookies, connection limits |
| L7 (application) | Resource exhaustion — expensive requests | HTTP flood, ReDoS, GraphQL nested queries | Rate limiting, WAF, query depth limits |

#### 6.2 Ransomware

Encrypts your data → demands payment → availability dies until paid (or restored from backups). Often **integrity** ka attack hota hai pehle (encrypt = modify), phir availability impact.

#### 6.3 Resource exhaustion (often self-inflicted)

- Unbounded queue → OOM
- Connection leak → DB pool exhausted
- Cache stampede → DB overwhelmed when cache expires
- Retry storms → upstream killed by aggressive client retries

#### 6.4 Single points of failure (SPOF)

Yeh architect ka favorite enemy hai. Audit every component:
- Single load balancer? SPOF.
- Single DB master? SPOF.
- Single region? SPOF.
- Single cloud provider? SPOF (rare but real — AWS us-east-1 outages have brought down half the internet).

### How availability is enforced

#### 6.1 Redundancy

```yaml
# Kubernetes example: 3 replicas across 3 zones
apiVersion: apps/v1
kind: Deployment
spec:
  replicas: 3
  template:
    spec:
      topologySpreadConstraints:
        - maxSkew: 1
          topologyKey: topology.kubernetes.io/zone
          whenUnsatisfiable: DoNotSchedule
```

#### 6.2 Rate limiting (every public endpoint)

```csharp
// .NET 8 rate limiting — protect from runaway clients and L7 DDoS
builder.Services.AddRateLimiter(opts =>
{
    opts.AddTokenBucketLimiter("api", o =>
    {
        o.TokenLimit = 100;
        o.TokensPerPeriod = 10;
        o.ReplenishmentPeriod = TimeSpan.FromSeconds(1);
        o.QueueLimit = 20;
    });
});
```

#### 6.3 Circuit breakers

```csharp
// Polly: stop calling a downstream that's already failing
var policy = Policy
    .Handle<HttpRequestException>()
    .CircuitBreakerAsync(
        exceptionsAllowedBeforeBreaking: 5,
        durationOfBreak: TimeSpan.FromSeconds(30)
    );
```

Without this: tera service downstream ke saath dub jata hai. Cascading failure = full outage.

#### 6.4 Graceful degradation

> Better to serve a stale cached response than no response. Better to disable comments than crash the whole site.

Feature flags + cached fallbacks + read-only modes — yeh "soft availability" engineering hai jo SREs aur architects share karte hain.

#### 6.5 Backup & restore (the forgotten pillar)

**Untested backup = no backup.** Practice restoration quarterly. Document RTO (Recovery Time Objective) and RPO (Recovery Point Objective).

| Metric | Question |
|---|---|
| **RTO** | How long to get back online? |
| **RPO** | How much data loss is acceptable? |

E-commerce RPO often **<5 minutes** (very fresh backups). Internal HR system RPO might be **24 hours**.

### Anti-patterns

| Anti-pattern | Why it's bad |
|---|---|
| "We have a backup" without restore test | Schrödinger's backup — both intact and corrupt until you check |
| Single region "for cost" | First regional outage = total outage |
| Synchronous calls to flaky third-party in request path | Their outage = your outage |
| No rate limiting on public endpoints | One bad actor with a loop kills you |
| No circuit breakers between microservices | Cascading failure inevitable |

### Backend-dev intuition

> Availability **engineering meets security** at the rate limiter, the WAF, and the backup policy. SRE aur SecOps overlap karte hain — yeh **interface zone** important hai.

## 7. The Trade-offs Triangle

CIA pillars often **conflict**. Architecture = choosing the right balance.

```
                  Confidentiality
                       /\
                      /  \
                     / X  \
                    /      \
                   /  TRADE \
                  /   -OFFS  \
                 /            \
                /______________\
           Integrity        Availability
```

### Real trade-off examples

| Decision | C ↑ but… | I ↑ but… | A ↑ but… |
|---|---|---|---|
| Encrypt every column | A: slower queries | — | — |
| Sign every API call | — | A: latency, complexity | — |
| Public read-only replica | C: more surface area to leak | — | A ✅ |
| Strict input validation | — | I ✅ | A: legitimate edge cases blocked |
| Aggressive rate limiting | — | — | A: blocks attackers but also burst users |
| Air-gapped offline backup | — | I ✅ (tamper-resistant) | A: slower to restore |
| Read replicas in multiple regions | C: data crosses borders → compliance risk | — | A ✅ |

### How an architect decides

There's no universal answer. Each system has a **threat model + business context** that ranks the pillars.

| System type | Likely priority order |
|---|---|
| Military classified network | C > I > A |
| Financial transactions (bank ledger) | I > C > A |
| E-commerce checkout | A > I > C |
| Healthcare EHR | C ≈ I > A (life-safety: A also high) |
| Industrial control system (SCADA) | I > A > C |
| Public news website | A > I > C |

> **Test yourself:** What's the order for **ShopFast's catalog page** vs **ShopFast's payment service**? (Hint: very different.)

## 8. Mental Models for the Architect

### Model 1 — "The Tripod"

CIA is a 3-legged stool. Weakest leg = how stable it is. **Investing 90% in C while ignoring A** still leaves you with a system that falls over.

### Model 2 — "Tag Every Requirement"

Habit drill: every new feature spec, write three lines:

```
Feature: User uploads profile photo
  C: M  (photos are user-private but not high-sensitivity)
  I: M  (tampering = embarrassing but not catastrophic)
  A: H  (broken upload = visible UX failure)
```

By Day 30, this becomes automatic. By Day 90, you do it during requirements review meetings — that's when you become "the security person on the team."

### Model 3 — "The Question Lens"

Before every architecture decision, ask:
1. **Who should see this?** (C)
2. **Who can change this?** (I)
3. **Who needs this to work?** (A)

Three questions. Every component. Every API. Every database row.

### Model 4 — "Threat → Pillar → Control"

When an incident happens, walk the chain:
- **Threat:** What did the adversary try?
- **Pillar:** Which CIA pillar did it target?
- **Control:** What control should have prevented or detected it?

This is the **post-mortem skeleton** for every security incident.

## 9. Beyond CIA — Parkerian Hexad & IAAA

CIA covers ~80% of cases. The remaining 20% needs more vocabulary.

### Parkerian Hexad (Donn Parker, 1998)

CIA + 3 more:

| Pillar | Meaning | Example |
|---|---|---|
| **Authenticity** | Source is genuine | Email is actually from your bank |
| **Possession / Control** | Authorized party has physical custody | USB drive containing data is in safe — even if encrypted, loss is a control breach |
| **Utility** | Data is in a usable form | Encrypted backup is intact (integrity ✅) but key is lost (utility ❌) |

Modern AI systems pressure CIA — when an LLM agent acts autonomously, **non-repudiation** (Authenticity-adjacent) becomes critical. "Did the agent really do this? Or was its prompt poisoned?"

### IAAA (the access management view)

When discussing identity & access, you'll hear:

| Letter | Meaning | We cover on |
|---|---|---|
| **Identification** | Claiming an identity ("I am alice") | Day 4 |
| **Authentication** | Proving it (password, MFA) | Days 61-67 |
| **Authorization** | What you can do | Days 75-81 |
| **Accountability/Auditing** | Recording what was done | Day 4 + throughout |

IAAA is the **operational mechanism** that delivers CIA. They're not competing frameworks — they're different views of the same elephant.

## 10. Architect's Decision Framework

Yeh tera daily tool ban jaye:

```
For each new feature / API / datastore:

  STEP 1 — Classify (CIA tagging):
          C: [H/M/L]   I: [H/M/L]   A: [H/M/L]

  STEP 2 — Justify (1 sentence each):
          Why this rating? What does failure look like?

  STEP 3 — Choose controls per pillar:
          C: Which access controls? Which encryption?
          I: Which validation? Which audit? Which versioning?
          A: Which redundancy? Which rate limit? Which fallback?

  STEP 4 — Identify trade-offs:
          What did I sacrifice? Is the business OK with that?

  STEP 5 — Document:
          ADR (Architecture Decision Record) entry.
          Future-you (and auditors) will thank you.
```

By Day 30 you'll be doing this in your head during sprint planning.

## 11. Deep Incident Study — Equifax (2017)

### The numbers

- **147 million Americans** (45% of US population), 15.2M UK citizens, 19,000 Canadians
- Data exposed: SSNs, DOBs, addresses, driver's license numbers, ~209,000 credit card numbers
- Settlement: **$700M+** (FTC + states + class action)
- CEO Richard Smith resigned, CSO Susan Mauldin resigned, CIO David Webb resigned
- Senior VP Jun Ying convicted of insider trading (sold stock before disclosure)
- Stock dropped ~35% in days of disclosure

### Timeline (compressed)

| Date | Event |
|---|---|
| **Mar 7, 2017** | Apache publishes Struts 2 patch for **CVE-2017-5638** (RCE in Jakarta Multipart parser) |
| **Mar 8** | Equifax security team notified — sends internal email to apply patch |
| **Mar 15** | Equifax scans for vulnerable systems — **scan misses the dispute portal** (config error) |
| **May 13** | Attackers exploit the unpatched Struts on the consumer dispute portal |
| **May 13 – Jul 30** | Attackers operate inside Equifax network for **76 days**, undetected |
| **Jul 29** | IDS finally detects suspicious traffic — discovery thanks to an expired SSL certificate that was re-installed, enabling traffic inspection that had been blind for **19 months** |
| **Jul 30** | Web application taken offline |
| **Sep 7** | Public disclosure (40 days after internal discovery) |

### The CVE explained

**CVE-2017-5638** — Apache Struts 2 Jakarta Multipart parser. If `Content-Type` header contained a crafted OGNL expression (Struts's expression language), the parser would evaluate it as code. Result: **remote code execution as the web server user, with no authentication required**.

Example malicious header:
```
Content-Type: %{(#_='multipart/form-data').(#dm=@ognl.OgnlContext@DEFAULT_MEMBER_ACCESS)...}
```

CVSS score: **10.0 (CRITICAL)**. Public exploit available within days of disclosure.

### CIA breakdown

| Pillar | What failed |
|---|---|
| **Confidentiality** ❌ | 147M records exfiltrated over 76 days. Egress monitoring effectively absent. |
| **Integrity** ❌ | Attackers had full RCE — could create accounts, modify data, plant persistence. Forensic gaps make full damage unknowable. |
| **Availability** ⚠️ | Front-end up, no DoS impact. Attackers explicitly avoided disruption to stay stealthy. (Stealth = sometimes A is "preserved" by attackers because outage triggers investigation.) |

### Root causes — multiple layers of failure

1. **Patch management broken** — patch existed for 2 months before exploit. Scanning tool missed the affected system due to misconfiguration. **No human verification.**
2. **Network segmentation absent** — dispute portal had direct database connectivity to 51 unrelated databases. **One foothold = data lake access.**
3. **Detective controls blind** — SSL inspection certificate had **expired 19 months prior**. Encrypted attacker traffic was invisible. Discovery only happened by accident after a renewal.
4. **Egress monitoring missing** — TBs of data left the network undetected. No DLP, no anomaly detection on outbound flows.
5. **Credentials in plaintext** — once inside, attackers found a file with **unencrypted admin credentials** that unlocked further databases.
6. **Audit logging insufficient** — full attacker timeline still partial; forensics could not establish all accessed records.
7. **Governance failure** — security team had reported patching gaps **annually** for years. Reports went unread by leadership.

### The architectural decisions that would have prevented it

| Failure | Preventive control |
|---|---|
| Missed patch | Automated CVE scanning + 72-hour SLA for critical patches + human verification |
| Lateral movement | Network microsegmentation — dispute portal can only reach dispute DB, not 51 others |
| Blind to encrypted traffic | Certificate lifecycle management (auto-rotation + monitoring); modern alternative: eBPF / sidecar inspection |
| Data exfiltration not detected | Egress DLP + anomaly detection on outbound volume |
| Plaintext credentials | Secrets manager (Vault, AWS Secrets Manager); no secrets on disk |
| Forensic gaps | Centralized SIEM with immutable storage; tamper-evident audit logs |
| Governance gap | Risk register reviewed at board level quarterly; security KPIs in CEO scorecard |

### The architect's takeaway

> Equifax was not breached by a **brilliant zero-day**. It was breached by **boring operational failures** — unpatched software, expired cert, no segmentation. **Discipline beats brilliance in security**, and architecture is how you encode discipline.

## 12. Pattern Recognition — Mini Case Studies

To train your eye, here are two more (very brief) breaches mapped to CIA. Notice the **pattern** — most major breaches are CIA failures stacked on each other.

### Mini case 1 — Capital One (2019)

- **What happened:** Misconfigured AWS WAF + over-privileged IAM role → former AWS employee exfiltrated 100M records.
- **CIA primary failure:** Confidentiality (massive data exposure).
- **Architectural lesson:** Least privilege on IAM roles + tighter SSRF protections + alerting on unusual S3 access patterns.

### Mini case 2 — NotPetya (2017, Maersk among many victims)

- **What happened:** Russian state actors weaponized a Ukrainian tax software update (M.E.Doc) → wiper malware spread globally → Maersk lost 49,000 laptops, 4,000 servers, $300M.
- **CIA primary failure:** Integrity (data wiped) → Availability (operations halted globally).
- **Architectural lesson:** Supply chain attestation, immutable backups, recovery exercises. Maersk recovered partly because **one domain controller** in Ghana happened to be offline during the attack (power outage = accidental air gap).

> See the pattern? Single-pillar failures are rare. **Real incidents cascade**. Tomorrow's vocabulary (Day 2) will give you words for these chains.

## 13. Common Misconceptions Cleared

| Misconception | Reality |
|---|---|
| "Encryption solves confidentiality" | Only data confidentiality — and only if keys are managed correctly. Doesn't help against authorized abuse. |
| "Backups solve availability" | Untested backups solve nothing. Plus, attackers target backups (ransomware-as-a-service routinely deletes backups first). |
| "HTTPS = secure" | HTTPS = encrypted in transit. Says nothing about server security, app logic, auth, storage. |
| "We're too small to be targeted" | Automated attacks don't care about your size. 43% of attacks target small businesses (Verizon DBIR). |
| "CIA is enough" | CIA is a starting point. Modern systems also need authenticity, non-repudiation, privacy, and resilience explicitly modeled. |
| "Security and usability are opposites" | Sometimes, but often false. Passkeys are MORE secure AND more usable than passwords. Good design dissolves the trade-off. |
| "Compliance = security" | Compliance is a floor, not a ceiling. Equifax was PCI-compliant. So was Target. |

## 14. Glossary

Terms introduced today — yeh tu kal use karega bhi aur sun-na bhi padega:

| Term | Meaning |
|---|---|
| **Adversary** | Anyone (or anything) acting against your security goals |
| **Threat actor** | Specific adversary (e.g., nation-state, criminal group, insider) |
| **Defense in depth** | Multiple independent layers of controls so single-layer failure isn't catastrophic |
| **Data at rest / in transit / in use** | Stored / moving across network / being processed in memory |
| **DLP** | Data Loss Prevention — controls detecting/blocking sensitive data exfiltration |
| **DDoS** | Distributed Denial of Service — many sources flooding a target |
| **HMAC** | Hash-based Message Authentication Code — proves integrity + authenticity with shared key |
| **IDOR** | Insecure Direct Object Reference — auth bypass by changing an ID in a request |
| **Non-repudiation** | Sender can't deny they sent/signed something (digital signatures provide this) |
| **PII** | Personally Identifiable Information |
| **RBAC / ABAC / ReBAC** | Role-/Attribute-/Relationship-based access control models (deep dive Days 75-81) |
| **RCE** | Remote Code Execution — attacker runs arbitrary code on your server |
| **RTO / RPO** | Recovery Time / Recovery Point Objective — how fast you recover / how much data loss is acceptable |
| **SIEM** | Security Information & Event Management — log aggregation + alerting platform |
| **SLA / SLO / SLI** | Service Level Agreement / Objective / Indicator — uptime contracts and measurements |
| **SPOF** | Single Point of Failure |
| **TLS** | Transport Layer Security — successor to SSL, protocol behind HTTPS |
| **WAF** | Web Application Firewall — L7 filter for HTTP attacks |
| **Zero-day** | Vulnerability unknown to the vendor (no patch available) |
