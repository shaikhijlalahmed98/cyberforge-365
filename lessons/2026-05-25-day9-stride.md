## 2. Core Concept

### 2.1 Pehle problem establish karte hain: open-ended brainstorm kyun fail hota hai

Chal kal wala scene yaad kar. Tune ShopFast checkout pe "what can go wrong?" socha. Shayad tujhe yeh threats aaye:

- "Koi cart total tamper kar de" ✅
- "Koi doosre ka order dekh le" ✅

Acha. Par ab imaandari se — kya tune **yeh** socha?

- Koi attacker payment gateway *ban ke* ShopFast ko fake "payment success" bhej de? (Spoofing)
- Koi user refund le ke baad claim kare "maine kabhi refund request hi nahi kiya" — aur tere paas proof na ho? (Repudiation)
- Koi checkout endpoint ko itna flood kar de ke real customers checkout hi na kar payein? (DoS)

Mushkil hai na? Yahi problem hai **open-ended brainstorm** ki. Insaani dimaag *availability bias* ka shikaar hota hai — woh sirf woh threats sochta hai jo recently dekhe/padhe. Jo category dimaag mein nahi hai, woh blind spot reh jaati hai. Aur security mein ek bhi blind spot = ek pura attack class jo undefended hai.

Software engineering mein iska solution hamesha ek raha hai: **checklist.** Pilot takeoff se pehle checklist chalata hai — chahe woh 10,000 ghante ka experienced ho — kyunki memory reliable nahi, checklist hai. Surgeon WHO surgical checklist chalata hai. Tu code review mein checklist follow karta hai (auth? input validation? error handling? tests?). **STRIDE security threats ka woh checklist hai.**

> **Backend-dev intuition:** Tu jab REST API design karta hai, tu mentally ek checklist chalata hai — "GET idempotent hai? POST validation hai? 401 vs 403 sahi hai? pagination hai?" STRIDE bilkul wahi hai, bas threats ke liye: har component pe 6 sawaal — "spoof ho sakta? tamper? repudiate? leak? DoS? escalate?" Checklist memory ko replace karti hai.

### 2.2 History: STRIDE kahan se aaya

Term ko properly introduce karte hain, kyunki backstory isse stick karwati hai. Late 1990s, Microsoft. Windows aur Office mein security holes ki barsaat ho rahi thi, aur ek core problem yeh thi ke developers ko *pata hi nahi tha kis cheez ke against design karein.* "Secure code likho" — par kis threat ke against? Koi shared vocabulary nahi thi.

1999 mein do Microsoft engineers — **Loren Kohnfelder** aur **Praerit Garg** — ne ek internal paper likha: *"The Threats to Our Products."* Unka insight simple par powerful tha: har security threat ko 6 buckets mein daala ja sakta hai, aur har bucket ek **desirable property ka ulta** hai. Agar tu chahta hai system mein 6 achhi properties hon, toh 6 tarah ke attacks unhe tod sakte hain. Un 6 buckets ke pehle letters se acronym bana: **STRIDE.**

Phir 2002 mein Bill Gates ka woh famous **"Trustworthy Computing" memo** aaya (Day 8 mein zikr kiya tha) — poori Microsoft ruk gayi taake security audit ho. Usi initiative mein **Adam Shostack** aur team ne STRIDE ko ek repeatable, teachable engineering practice banaya — diagrams pe apply karne layak. Aaj STRIDE threat modeling ka *de facto* starting framework hai — Microsoft SDL, OWASP, har serious shop mein.

STRIDE ka magic yahi "negation" wala idea hai. Dekh:

| STRIDE Threat | Jo property todta hai | Day mein kahan mila |
|---------------|----------------------|---------------------|
| **S**poofing | Authentication | Day 4 (AuthN) |
| **T**ampering | Integrity | Day 1 (CIA - I) |
| **R**epudiation | Non-repudiation / Accountability | Day 4 (3rd A - accounting) |
| **I**nformation Disclosure | Confidentiality | Day 1 (CIA - C) |
| **D**enial of Service | Availability | Day 1 (CIA - A) |
| **E**levation of Privilege | Authorization | Day 4 (AuthZ) |

> **Mental model #1 (carry forward):** *STRIDE = teri desirable security properties, ulti karke.* Confidentiality, Integrity, Availability (CIA) + Authentication, Authorization, Non-repudiation (AAA) — in 6 properties ka negation hi STRIDE ke 6 letters hain. Agar tujhe STRIDE bhool jaye, CIA+AAA se reverse-engineer kar le.

### 2.3 Har letter — story, ShopFast example, mitigation

Ab ek-ek letter ko kahani ke saath kholte hain. Har ek ke liye: **kya hai → ek real shakal → ShopFast pe → kaise roke (.NET/Spring).**

#### S — Spoofing (Authentication ka dushman)

**Spoofing** ka matlab: kisi *aur* hone ka dawa karna — ek identity ko *naqli* taur pe pehenna. Tu jo ho woh nahi, par system ko convince kar dena ke tu woh hai.

Story se samajh: jab tu login karta hai, server ko kaise pata ke "tu Ijlal hai"? Woh tere password/token pe bharosa karta hai. Spoofing tab hota hai jab attacker us proof ko **chura, guess, ya bypass** kar leta hai — phir woh "Ijlal ban ke" system se baat karta hai. Yeh sirf users ke saath nahi — **machines bhi spoof hoti hain.** ShopFast ka Checkout Service jab Payment Gateway se baat karta hai, woh kaise jaanta ke woh *asli* gateway hai, koi attacker ka fake server nahi? Agar woh proof nahi maangta — spoofing possible.

**ShopFast example:** Attacker Payment Gateway ki taraf se ek **fake "payment confirmed" webhook** bhej deta hai. Agar Checkout Service us callback ko verify nahi karta (signature/mTLS), toh order ship ho jaata bina paise aaye. Yeh service-to-service spoofing hai.

**Mitigation (canonical):** **Authentication** — strong identity proof. Users ke liye: passwords + MFA, secure session tokens. Machines ke liye: mutual TLS (mTLS), signed webhooks (HMAC), API keys/OAuth client creds.

```csharp
// ✅ Anti-spoofing: payment webhook ki signature verify karo (HMAC)
[HttpPost("api/webhooks/payment")]
[AllowAnonymous] // gateway authenticate nahi hota — signature se prove karta hai
public IActionResult PaymentCallback([FromBody] string rawBody,
                                     [FromHeader(Name = "X-Pay-Signature")] string sig)
{
    var expected = HmacSha256(rawBody, _secret); // sirf asli gateway ke paas secret hai
    if (!CryptographicOperations.FixedTimeEquals(   // timing-safe compare (Day 2 recall: side-channel)
            Encoding.UTF8.GetBytes(sig),
            Encoding.UTF8.GetBytes(expected)))
        return Unauthorized();                       // spoofed caller → reject
    // ... ab bharosa kar sakte hain ke yeh asli gateway hai
    return Ok();
}
```

#### T — Tampering (Integrity ka dushman)

**Tampering** = data ya code ko *unauthorized* taur pe **badalna**. Day 1 yaad kar — "padh nahi sakta par change kar sakta hai" wala scenario integrity ka dushman hai. Tampering exactly woh hai.

Story: data do haalaton mein vulnerable hota hai — **in transit** (network pe travel karte waqt) aur **at rest** (database/file mein pada). Agar attacker dono mein se kahin bhi value badal de bina kisi ko pata chale — tampering. Aur sabse khatarnaak tampering woh hai jo *client se aane wale data* pe trust karne se hoti hai (Day 8 ka refund example).

**ShopFast example:** Client checkout request mein `cartTotal: 1.00` bhej deta hai jabke asli total ₹5000 hai. Agar Checkout Service us client value pe trust karke charge kar de — integrity tampered, attacker ₹1 mein saaman le gaya.

**Mitigation:** **Integrity controls** — server-side recompute (client value ko authority mat do), input validation, hashing/digital signatures (data ki authenticity), TLS (in transit), HMAC, database integrity constraints, immutable audit logs.

```java
// ✅ Anti-tampering: total SERVER compute karta hai, client ka bheja IGNORE
@PostMapping("/api/cart/checkout")
public ResponseEntity<OrderDto> checkout(@RequestBody CheckoutRequest req, Authentication auth) {
    Cart cart = carts.loadForUser(((AppUser) auth.getPrincipal()).getId(), req.getCartId());
    // client ka req.getClientTotal() ko CHHUA tak nahi — server source of truth
    BigDecimal total = pricing.computeTotal(cart);   // catalog se asli prices
    Order order = orders.create(cart, total);
    return ResponseEntity.ok(OrderDto.from(order));
}
```

#### R — Repudiation (Non-repudiation ka dushman)

**Repudiation** = koi ek action karke baad mein **mukar jaye**, aur system saabit na kar paye ke usne kiya. Yeh Day 4 ka woh "teesra, under-loved A" — accounting/non-repudiation — ka dushman hai.

Story se: imagine ek admin ShopFast pe ek bada refund issue karta hai apne dost ko, phir baad mein kehta hai "maine nahi kiya, shayad bug hoga." Agar tere paas **tamper-evident log** nahi jo kahe "userId=42 ne 2026-05-25 14:03 pe refund #999 banaya, IP x.x.x.x se" — toh tu yeh dawa challenge nahi kar sakta. Woh action ko *repudiate* kar gaya. Insider threat (Day 4 ka Morgan Stanley case) ka yeh dil hai.

**ShopFast example:** Ek warehouse staff inventory adjustment karke stock chura leta hai, aur logs mein sirf "stock changed" likha hai — *kisne* nahi. Forensics impossible. Repudiation win.

**Mitigation:** **Non-repudiation controls** — secure, append-only audit logs (kaun/kya/kab/kahan), digital signatures, timestamps, log integrity protection (taake attacker apne traces mita na sake — yeh khaas important hai).

```csharp
// ✅ Anti-repudiation: har sensitive action ka tamper-evident record
await _audit.Record(new AuditEvent {
    Actor = User.GetUserId(),          // kaun
    Action = "refund.create",          // kya
    Target = $"order:{orderId}",       // kis pe
    Amount = refundable,
    At = DateTimeOffset.UtcNow,        // kab (UTC, server clock)
    SourceIp = HttpContext.Connection.RemoteIpAddress?.ToString(), // kahan se
    // logs append-only store mein (e.g. WORM bucket / signed) — actor mita na sake
});
```

#### I — Information Disclosure (Confidentiality ka dushman)

**Information Disclosure** = data un logon ko mil jaye jinhe nahi milna chahiye. Day 1 ka Confidentiality pillar ka seedha dushman.

Story: data leak hone ke laakhon raaste hain — unencrypted transmission, verbose error messages jo stack trace + DB schema ugal dete hain, ek API jo zyada fields return karti hai (over-fetching), ek S3 bucket jo public reh gaya, ya ek endpoint jo authZ check bhool gaya. Yeh STRIDE ka sabse "common" letter hai — zyadatar breaches isi mein girte hain.

**ShopFast example:** `GET /api/orders/{id}` order ke saath poora `customer` object return karta hai jisme email, phone, address — chahe woh dusre user ka ho (IDOR — Day 3). Ya ek error message client ko `SQLException: column 'card_last4' ...` dikha deta hai.

**Mitigation:** **Confidentiality controls** — encryption (at rest + in transit), authorization checks, least-privilege data access, DTOs (sirf zaroori fields return), generic error messages (details server log mein, client ko nahi), masking/redaction.

```java
// ❌ Information disclosure: entity seedha return — saare fields leak
return order; // includes customer PII, internal flags, cost price...

// ✅ DTO se sirf zaroori fields — leak surface minimal
return OrderResponse.builder()
    .id(order.getId())
    .status(order.getStatus())
    .total(order.getTotal())
    .build(); // PII, internal cost, flags — kuch leak nahi hua
```

#### D — Denial of Service (Availability ka dushman)

**Denial of Service (DoS)** = system ko itna busy/crash kar dena ke *legitimate* users use na kar payein. Day 1 ka Availability pillar ka dushman. Yaad rakh — yeh sirf "bandwidth flood" nahi; ek **single expensive request** bhi service gira sakti hai (algorithmic DoS).

Story: DoS ke teen flavours hain — **volumetric** (bohot saari traffic se pipe bhar do), **protocol** (TCP/connection state exhaust karo), aur **application-layer** (ek aisi request jo server pe bohot kaam karwaye — e.g. ek search query jo poora DB scan kare, ya ek regex jo exponential time le — "ReDoS"). Backend dev ke liye sabse insidious yeh application-layer wala hai, kyunki traffic "normal" lagti hai.

**ShopFast example:** Attacker checkout endpoint ko ek aisi cart ke saath hit karta hai jisme 1 million line-items hain — har item pe price lookup → DB melt. Ya coupon-validation ka regex catastrophic backtracking mein chala jaata hai. 10 requests = service down. Real customers checkout nahi kar pa rahe = direct revenue loss.

**Mitigation:** **Availability controls** — rate limiting / throttling, input size limits (max cart items), quotas, timeouts, circuit breakers (tujhe yeh microservices se pata hai!), autoscaling, CDN/WAF/DDoS scrubbing (Cloudflare, AWS Shield), resource caps.

```csharp
// ✅ Anti-DoS: rate limit + input bound (ASP.NET Core rate limiting)
builder.Services.AddRateLimiter(o => o.AddFixedWindowLimiter("checkout", opt => {
    opt.PermitLimit = 10;                 // per user/IP
    opt.Window = TimeSpan.FromMinutes(1);
}));

// aur application-layer guard:
if (cart.Items.Count > 500)               // bounded input — million-item cart reject
    return BadRequest("Cart too large");
```

#### E — Elevation of Privilege (Authorization ka dushman)

**Elevation of Privilege (EoP)** = ek kam-privilege user/process apne aap ko zyada-privilege bana le. Day 4 ka Authorization (AuthZ) ka dushman. Yeh STRIDE ka sabse **khatarnaak** letter hai — kyunki EoP usually baaki saare controls bypass kar deta hai (admin ban gaya toh sab kar sakta hai).

Story: do tarah ki EoP hoti hai — **vertical** (normal user → admin) aur **horizontal** (user A → user B ka data, same privilege level par "doosra banना" — yeh IDOR hai). Classic EoP: ek hidden admin endpoint jo authZ check bhool gaya; ya ek JWT jisme `role` claim client-side editable hai; ya Day 7 ka Capital One — SSRF se IMDS, phir over-broad IAM role assume kar ke poora S3.

**ShopFast example:** Normal customer `POST /api/admin/refunds/approve` ko call karta hai aur woh endpoint sirf authentication check karta hai (logged in?) par authorization nahi (admin?) — customer khud apne refunds approve karne lagta hai. Vertical EoP.

**Mitigation:** **Authorization controls** — har sensitive operation pe explicit authZ check (RBAC/ABAC), server-side role enforcement (kabhi client claim pe trust nahi), least privilege, deny-by-default, sandboxing.

```java
// ✅ Anti-EoP: method-level authZ, deny-by-default
@PreAuthorize("hasRole('REFUND_APPROVER')")   // role server-side enforce
@PostMapping("/api/admin/refunds/{id}/approve")
public ResponseEntity<Void> approve(@PathVariable Long id, Authentication auth) {
    // sirf REFUND_APPROVER yahan pohoch sakta — customer ko 403
    refunds.approve(id, ((AppUser) auth.getPrincipal()).getId());
    return ResponseEntity.ok().build();
}
```

> **Backend-dev intuition:** Gaur kar — har STRIDE letter ki mitigation woh hi cheezein hain jo tu *already* "good engineering" ke naam pe karta hai (validation, auth, logging, rate limiting). STRIDE naya kaam nahi maangta — woh **guarantee** maangta hai ke tune har category socha. Discipline, naya tool nahi.

### 2.4 STRIDE-per-element — kis cheez pe kaunse threats

Ab ek level upar. Tu pooch sakta hai: "har component pe saare 6 letters apply karun?" Microsoft ne ek shortcut diya jo time bachata hai: **STRIDE-per-element.** Idea — har threat har *type* ke component pe nahi lagta. Apne system ko 4 building blocks mein todo (yeh Day 10 ka Data Flow Diagram preview hai):

- **External entity** (user, 3rd-party service — tere control ke bahar)
- **Process** (tera code — service, API, function)
- **Data store** (DB, file, cache, queue)
- **Data flow** (ek line — data jo do cheezon ke beech move karti hai)

Microsoft ki classic applicability table (yeh yaad rakhne layak hai):

| Element | S | T | R | I | D | E |
|---------|---|---|---|---|---|---|
| **External entity** (user/3rd-party) | ✅ | | ✅ | | | |
| **Process** (your service) | ✅ | ✅ | ✅ | ✅ | ✅ | ✅ |
| **Data store** (DB/cache/file) | | ✅ | ✅* | ✅ | ✅ | |
| **Data flow** (network/IPC) | | ✅ | | ✅ | ✅ | |

*(\* data store pe repudiation tab relevant jab woh khud log/audit store ho.)*

Padhne ka tareeqa: ek **process** (jaise Checkout Service) sab kuch face karta hai — woh spoof ho sakta, tamper, repudiate, leak, DoS, escalate. Lekin ek **data flow** (network line) pe sirf Tampering, Info Disclosure, DoS sochna kaafi (us line ko spoof/escalate nahi karte — woh endpoints ka kaam hai). Yeh table tujhe **systematic** banata hai: har element pe sirf relevant letters chala, randomly nahi.

> **Mental model #2:** *STRIDE-per-element = "har box/line pe sirf uske applicable letters chala."* External entity = mostly S+R; data flow = mostly T+I+D; process = poora STRIDE. Yeh checklist ko thakawat se bachata hai aur coverage guarantee karta hai.

Ek aur variant hai **STRIDE-per-interaction** (Shostack ka refinement) — jahan tu har *interaction* (source → flow → destination) pe STRIDE chalata hai. Yeh zyada precise par zyada time-consuming. Beginner ke liye STRIDE-per-element kaafi; enterprise threat models per-interaction use karte hain.

### 2.5 STRIDE → mitigation map (yaad rakhne wala table)

STRIDE ki sabse badi practical value: har letter ka ek **canonical defense category** hai. Threat pakda → mitigation type pata. Yeh table interview aur design dono mein kaam aata hai:

| Threat | Property violated | Canonical mitigation | Concrete examples |
|--------|-------------------|----------------------|-------------------|
| Spoofing | Authentication | **Authenticate** | MFA, mTLS, signed tokens, HMAC webhooks |
| Tampering | Integrity | **Protect integrity** | Server-side recompute, signatures, TLS, input validation, hashes |
| Repudiation | Non-repudiation | **Log + sign** | Append-only audit logs, timestamps, digital signatures |
| Information Disclosure | Confidentiality | **Encrypt + authorize** | TLS, encryption at rest, DTOs, least-privilege, generic errors |
| Denial of Service | Availability | **Limit + isolate** | Rate limits, quotas, timeouts, circuit breakers, autoscale, CDN |
| Elevation of Privilege | Authorization | **Authorize + isolate** | RBAC/ABAC, deny-by-default, least privilege, sandboxing |

### 2.6 Code: ek ShopFast endpoint ko poore STRIDE se guzaarna

Chal Day 8 wala refund endpoint le ke usey **systematically** STRIDE-check karte hain. Yahi asli skill hai — ek endpoint, 6 sawaal:

```csharp
[HttpPost("api/orders/{orderId}/refund")]
[Authorize]                                    // baseline authN
public async Task<IActionResult> CreateRefund(long orderId) { ... }
```

| Letter | Sawaal is endpoint pe | Mitigation yahan |
|--------|----------------------|------------------|
| **S** | Koi aur user ban ke call kar sakta? Token chura/forge? | `[Authorize]` + strong session tokens + MFA on sensitive ops |
| **T** | Refund amount client se aata? orderId tamper? | Amount **server compute** kare; ownership server-side verify |
| **R** | Refund hua, user mukar jaye — proof hai? | Append-only `_audit.Record(actor, action, target, at, ip)` |
| **I** | Response dusre order/PII leak karta? Error verbose? | DTO; ownership check; generic errors |
| **D** | Refund endpoint flood ho sakta? | Rate limit per-user; idempotency key (double-refund se bachao) |
| **E** | Normal user dusre ka/admin refund kar sakta? | Ownership + `[Authorize(Roles="...")]` for admin path; deny-by-default |

Dekha? Kal humne is endpoint mein **2 design flaws** "instinct" se pakde the (Tampering + Broken Access Control). Aaj STRIDE ne **6 angles** systematically diye — aur usme **Repudiation** aur **DoS** woh the jo instinct mein miss ho gaye the. **Yahi STRIDE ki poori value hai: woh tujhe woh threats yaad dilata hai jo tu apne aap nahi sochta.**

### 2.7 STRIDE kab kaafi nahi — limits

STRIDE shaandar hai par sab-kuch nahi. Imaandar architect uski limits jaanta hai:

1. **Privacy threats poori tarah cover nahi.** STRIDE security pe focus karta hai (CIA+AAA), lekin "data subject ka consent," "linkability," "data retention" jaise privacy threats ke liye ek alag framework hai — **LINDDUN** (Day 9+ futurist, aur Month 9 mein detail). GDPR-heavy systems ke liye STRIDE + LINDDUN dono chahiye.

2. **AI/ML threats fit nahi baithte.** Prompt injection, model poisoning, training-data extraction — yeh STRIDE ke 6 buckets mein clean nahi girte. Inke liye **OWASP LLM Top 10** aur MITRE **ATLAS** hain.

3. **Business-logic abuse miss ho sakti.** "Koi 1000 fake accounts banaake referral bonus farm kare" — yeh technically koi STRIDE letter nahi todta, par business ko nuksaan. STRIDE ko **abuse-case thinking** ke saath complement karo.

4. **Severity nahi deta.** STRIDE batata hai "kya galat ho sakta" — *kitna bura* nahi. Prioritization ke liye **DREAD** (Day 12) ya CVSS chahiye.

> **Architect's honest take:** STRIDE ek **starting checklist** hai, final answer nahi. Woh tujhe 80% common threats systematically dila deta hai — baaki 20% (privacy, AI, business logic) ke liye specialized lenses chahiye. Ek tool se shaadi mat kar.

## 3. Mental Models

1. **STRIDE = security properties, ulti** — CIA (Confidentiality, Integrity, Availability) + AuthN, AuthZ, Non-repudiation ka negation. Bhool jaye toh in 6 properties se reverse karo.

2. **STRIDE-per-element** — har box/line pe sirf uske applicable letters chala (external entity = S+R; data flow = T+I+D; process = poora STRIDE). Coverage guarantee, thakawat nahi.

3. **Threat → canonical mitigation** — har letter ka ek default defense category hai (Spoofing→authenticate, Tampering→integrity-protect, R→log, I→encrypt+authorize, D→rate-limit, E→authorize). Threat pakda = mitigation type pata.

4. **STRIDE answers Shostack Q2** — Day 8 ka "what can go wrong?" ka structured engine. STRIDE akela nahi chalta — woh 4-question loop ke andar baithta hai.

## 4. Trade-offs Analysis

| Dimension | Open-ended brainstorm | STRIDE checklist | STRIDE-per-interaction |
|-----------|----------------------|------------------|------------------------|
| **Coverage (missed categories?)** | ❌ High miss rate (bias) | ✅ 6 categories guaranteed | ✅✅ Per data-flow precision |
| **Time cost** | Low (par incomplete) | Moderate | High |
| **Skill needed** | High (expert intuition) | Low-moderate (teachable) | Moderate-high |
| **Best for** | Quick gut-check | Most teams, most features | Enterprise / high-risk systems |
| **Risk** | Blind spots | Can feel mechanical; misses privacy/AI/biz-logic | Analysis paralysis on big diagrams |
| **Onboarding new devs** | ❌ Hard to teach | ✅ Easy (6 letters) | Moderate |

**Architect's read:** STRIDE ka asli tohfa **democratization** hai — woh threat modeling ko "security expert ka jaadu" se "har dev ki teachable skill" bana deta hai. Yeh shift-left (Day 8) ko culturally possible banata hai. Trade-off: woh thoda mechanical lag sakta hai aur privacy/AI/business-logic threats miss karta hai — isliye usse *starting point* maano, gospel nahi.

## 5. Architect's Decision Framework

Jab kisi feature/system pe STRIDE chalana ho:

1. **Q1 pehle (Day 8 loop)** — System ka simple diagram banao. Components ko 4 types mein todo: external entities, processes, data stores, data flows. (Day 10: proper DFD.)

2. **Har element pe applicable letters chalao** — STRIDE-per-element table use karo (§2.4). Process = poora STRIDE; data flow = T/I/D; external entity = S/R. Har applicable letter pe ek concrete sawaal pooch.

3. **Threats likho, generic nahi** — "Tampering possible" ❌. "Client checkout total tamper karke ₹1 mein order" ✅. Specific threat = actionable.

4. **Har real threat ko mitigation map se jodo** — §2.5 table se canonical defense uthao, phir Week 1 ke control-type (preventive/detective/corrective) label lagao.

5. **Prioritize karo** — sab threats equal nahi. Abhi gut-feel (likelihood × impact); Day 12 mein DREAD formalize karega. Top threats pehle.

6. **Limits yaad rakho** — privacy-heavy? LINDDUN add karo. AI component? OWASP LLM Top 10. Business logic? Abuse cases. STRIDE 80% deta hai, 100% nahi.

7. **Feed forward** — threats → risk register (Day 7) → controls → backlog tickets. Threat model exercise nahi, real engineering work generate karta hai.

## 7. Real-World Incident — Zerologon CVE-2020-1472 (deep study)

> **Lesson tie:** Zerologon ek *textbook STRIDE* incident hai — ek **cryptographic design flaw** (Day 8 ka "design flaw, not impl bug") jo do STRIDE letters ek saath todta hai: **Spoofing** (Domain Controller ki identity naqli banao) aur **Elevation of Privilege** (unauthenticated attacker → Domain Admin). CVSS score: **10.0 — maximum possible.**

### Background — scene set karte hain

Enterprise networks Windows **Active Directory (AD)** pe chalte hain — woh central "identity ka boss" jo decide karta hai kaun kaunse machine/resource pe ja sakta. AD ka dil hai **Domain Controller (DC)** — woh server jo authentication handle karta hai. Computers (aur servers) DC se baat karne ke liye ek protocol use karte hain: **Netlogon Remote Protocol (MS-NRPC)** — yeh woh handshake hai jahan ek machine apne aap ko DC ke saamne prove karti hai.

Ab story ka technical dil. Netlogon ke authentication handshake mein ek cryptographic step tha jahan client aur DC ek shared session key se ek value encrypt karte the — Microsoft ne yeh **AES in CFB8 mode** se kiya. AES-CFB8 ko securely chalane ke liye ek **Initialization Vector (IV)** chahiye jo har baar **random/unique** ho (IV = ek starting random value jo same plaintext ko har baar alag ciphertext banata hai). **Microsoft ki galti:** unhone IV ko **hardcode karke saara zero (all-zero, 16 bytes of 0x00)** set kar diya. Yeh ek design-level crypto blunder tha.

### Technical root cause — kyun yeh kaam kar gaya

Researcher **Tom Tervoort (Secura)** ne September 2020 mein paaya: jab IV all-zero ho AES-CFB8 mein, toh ek specific plaintext (all-zero bytes) ke liye, ciphertext bhi all-zero **256 mein se 1 baar (probability 1/256)** nikalta hai. Matlab attacker ko sirf **all-zero ciphertext bhej-bhej ke** ~256 baar try karna tha (average) — aur ek attempt successfully "authenticate" ho jaata bina koi password/key jaane. Yeh **Spoofing** hai: attacker DC ke saamne kisi bhi computer account (including DC khud) hone ka *naqli* proof de deta hai.

Aur yahin nahi rukta. Ek extra Netlogon call (`NetrServerPasswordSet2`) se attacker DC ke **apne computer account ka password khaali (empty) set** kar sakta tha. Iske baad woh DC ki credentials se replication kar ke **poore domain ke saare password hashes** (NTDS.dit) nikaal sakta tha — yaani **Domain Admin** ban jaata. Yeh **Elevation of Privilege** hai, network access se seedha "poore enterprise ka malik."

Sabse darawni baat: yeh attack **unauthenticated** tha (login ki zaroorat nahi), sirf DC tak **network reachability** chahiye thi, aur **seconds** mein chalta tha. Koi malware nahi, sirf protocol abuse.

### Numbers & timeline

| Date | Event |
|------|-------|
| Sep 2019 | Microsoft ne (quietly) ek partial Netlogon fix shuru kiya tha — pehle exact threat samjhe bina |
| Aug 11, 2020 | Microsoft ne patch jaari kiya (CVE-2020-1472), bina full detail ke |
| Sep 11, 2020 | Tom Tervoort (Secura) ne whitepaper + "Zerologon" naam release kiya — full technical detail |
| Sep 14, 2020 | Public PoC exploits GitHub pe aa gaye (Mimikatz, Impacket modules) |
| Sep 18, 2020 | **CISA Emergency Directive 20-04** — saari US federal agencies ko **4 din** mein patch karne ka order (yeh CISA rarely karta — severity ka indicator) |
| Feb 2021 | Microsoft ne "enforcement mode" by default on kiya (do-phase rollout) |

- **CVSS v3.1: 10.0** (maximum). Attack complexity low, no privileges, no user interaction.
- Crores Windows domains worldwide exposed; weeks ke andar active exploitation observed (ransomware gangs ne adopt kiya — Ryuk/Conti operators).

### STRIDE breakdown (aaj ka lens)

- **Spoofing (S):** All-zero IV ne attacker ko DC/computer account hone ka naqli authentication proof banane diya — identity spoofed, koi key jaane bagair.
- **Elevation of Privilege (E):** Unauthenticated network attacker → Domain Admin (poore AD ka control). STRIDE ka sabse khatarnaak combo.
- **Tampering (T):** Computer account ka password attacker ne empty set kiya — directory data ka unauthorized change.
- **Information Disclosure (I) — downstream:** Domain Admin ban ke saare password hashes (NTDS.dit) nikaal liye.

Ek hi crypto design flaw, **4 STRIDE letters** cascade. Yahi reason hai STRIDE checklist powerful hai — ek root flaw multiple categories trigger karta hai.

### Multiple root-cause layers (sirf "ek bug" nahi)

1. **Cryptographic design flaw (the real root):** Hardcoded all-zero IV in AES-CFB8 — yeh classic "crypto ko galat use karna" hai (Day 8: design flaw, na ki implementation typo). Koi crypto-aware threat model "IV unique/random hai kya?" pooch ke isse pakad leta.
2. **Custom/legacy protocol:** Netlogon ka yeh crypto Microsoft ka apna purana custom design tha, standard authenticated-encryption (jaise AES-GCM) nahi — "roll your own crypto" anti-pattern.
3. **No defense-in-depth on the DC handshake:** Ek hi check fail hua aur seedha Domain Admin — koi layered check nahi (Day 5 recall).
4. **Network exposure:** DC ka Netlogon RPC widely reachable tha andar ke network se — segmentation hoti toh blast radius kam (Day 3).

### Architectural controls jo har layer rokte

| Root-cause layer | Control jo rokta |
|------------------|------------------|
| All-zero IV crypto flaw | **Crypto design review / threat model** — "har IV unique & random hai?"; standard AEAD (AES-GCM) use karo, custom mode nahi |
| Roll-your-own protocol crypto | **Use vetted standards** — apna crypto mat likho; reviewed libraries |
| One check → Domain Admin | **Defense-in-depth** — tiered admin model, DC isolation, privileged access workstations |
| Wide network reachability | **Segmentation** — DC management plane restricted (Day 3 blast radius) |
| Detection | **Monitor Netlogon anomalies** (4742/5805 events, sudden machine-account password changes) — detective control (Day 6) |

### Architect's takeaway (quotable)

> *"Zerologon ek line ka crypto design choice tha — 'IV ko zero rakh do' — jo poore enterprise ko unauthenticated attacker ke haath mein de deta. Ek STRIDE-aware design review jo sirf yeh poochta — 'is authentication step ko spoof kaise kiya ja sakta hai? IV unique hai?' — isse code likhne se pehle pakad leta. Crypto mein 'thoda sa shortcut' = total compromise. Standards use karo, apna crypto mat banao."*

## 8. Pattern Recognition — Heartbleed CVE-2014-0160 (Information Disclosure)

Ek alag STRIDE letter ka clean example, taake breadth aaye. **April 2014:** OpenSSL — woh library jo aadhe internet ka HTTPS chalati hai — mein ek bug nikla jise **Heartbleed** kaha gaya. TLS ka ek "heartbeat" feature (connection alive rakhne ke liye chhota ping-pong) tha jahan client kehta "yeh 4-byte message wapas bhej" — par OpenSSL **client ke bheje length value pe trust** kar leta tha bina verify kiye ke actual message itna bada hai ya nahi. Attacker keh deta "yeh message 64KB bada hai" jabke woh 4 byte bhejta — server seedha apni memory se **64KB extra** padh ke wapas bhej deta, jisme private keys, session tokens, passwords, anyone's data ho sakta tha.

- **STRIDE category:** Pure **Information Disclosure** (Confidentiality) — koi tampering/EoP nahi, sirf memory leak. ~17% internet's secure web servers affected; private TLS keys chori ho sakti thi (matlab future traffic decrypt — "harvest now" ka classic).
- **Root cause:** **Missing input validation** — client-supplied length pe trust (yeh **Tampering→Info Disclosure** chain bhi hai: client ne length tamper ki). Classic Day 8 "client pe trust mat karo" + STRIDE "Tampering" ka real shakal.
- **Common pattern (Zerologon + Heartbleed):** Dono mein ek chhoti si galat assumption — "IV zero chalega" / "client ka length sahi hoga" — ne ek poora STRIDE category khol diya. **STRIDE ka checklist isi liye hai: woh in chhoti assumptions ko har category mein systematically challenge karta hai.**

## 9. Common Misconceptions Cleared

| Misconception | Reality |
|---------------|---------|
| "STRIDE ek tool/software hai" | STRIDE ek *mnemonic checklist* hai (6 categories). Tools (MS TM Tool, pytm) usse implement karte hain, par STRIDE khud soch ka framework hai. |
| "Har element pe saare 6 letters lagao" | Nahi — STRIDE-per-element: external entity = S+R, data flow = T+I+D, process = poora. Applicability table use karo. |
| "STRIDE severity/priority deta hai" | Nahi — woh sirf *kya galat ho sakta* batata hai. Priority ke liye DREAD (Day 12)/CVSS. |
| "STRIDE sab threats cover karta hai" | Nahi — privacy (LINDDUN), AI (ATLAS/LLM Top10), business-logic abuse miss karta. 80% tool hai. |
| "Spoofing aur EoP same cheez hai" | Nahi — Spoofing = *kisi aur ki identity pehenna* (authN break). EoP = *apni identity pe zyada power lena* (authZ break). |
| "Repudiation chhoti baat hai" | Insider threats aur forensics ka dil — bina audit logs ke tu attack saabit nahi kar sakta (Day 4 Morgan Stanley). |
| "DoS matlab sirf traffic flood" | Application-layer DoS (ek expensive request, ReDoS) zyada insidious — traffic 'normal' lagti hai. |

## 10. Glossary

- **STRIDE:** Microsoft ka threat-classification mnemonic — Spoofing, Tampering, Repudiation, Information disclosure, Denial of service, Elevation of privilege. Har letter ek security property ka negation.
- **Spoofing:** Kisi aur ki identity naqli taur pe pehenna; Authentication ka dushman.
- **Tampering:** Data/code ka unauthorized change (in transit ya at rest); Integrity ka dushman.
- **Repudiation:** Action karke mukar jaana jab system saabit na kar paye; Non-repudiation/accountability ka dushman.
- **Information Disclosure:** Data un tak pohonchna jinhe nahi milna chahiye; Confidentiality ka dushman.
- **Denial of Service (DoS):** Legitimate users ko service se rok dena (flood ya resource exhaustion); Availability ka dushman.
- **Elevation of Privilege (EoP):** Kam-privilege se zyada-privilege ban jaana (vertical: user→admin; horizontal: IDOR); Authorization ka dushman.
- **STRIDE-per-element:** Har DFD element type (external entity/process/data store/data flow) pe sirf applicable STRIDE letters chalane ka method.
- **STRIDE-per-interaction:** Har source→flow→destination interaction pe STRIDE chalane ka (zyada precise) variant.
- **Data Flow Diagram (DFD):** System ka diagram — external entities, processes, data stores, data flows + trust boundaries. (Day 10.)
- **Trust boundary:** Woh line jahan trust level badalta hai (e.g. browser→server). (Day 11.)
- **Initialization Vector (IV):** Crypto mein ek starting value jo har encryption ko unique banata hai; unique/random hona zaroori (Zerologon mein zero tha — flaw).
- **AEAD:** Authenticated Encryption with Associated Data (e.g. AES-GCM) — modern standard jo encryption + integrity ek saath deta; "roll your own crypto" ka safe alternative.
- **LINDDUN:** Privacy-focused threat modeling framework (STRIDE ka privacy cousin).
- **MITRE ATLAS / OWASP LLM Top 10:** AI/ML systems ke threats ke frameworks ("STRIDE for AI").
